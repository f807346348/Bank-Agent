AI-Bank Multiple Agent 工作流架构图

一个主 Agent 负责调度，五个专用 Agent 分别处理需求、SQL、数据访问、结果审核和 HTML 报告交付。 目标是跑通“业务报表需求到可打开报告”的最小闭环。

Demo 当前定位 验证多 Agent 分工是否能串起报表理解、只读查询、审核反馈和报告生成。

1. 架构关系

Main Agent 是总调度员，不直接写 SQL、不直接查库、不直接撰写报告。

输入

业务人员的自然语言报表需求

例如：近 12 个月各城市客户交易分析报告。

Main Agent

主对话调度 Agent

接收请求，按顺序调用各角色 Agent，传递上一步产物，并根据审核结论决定继续、重试或停止。

只做流程编排
保存必要摘要
最终写入 reports/
输出

可打开的 HTML 数据报告

包含标题、指标卡、表格、图表和自动分析结论。

01

业务需求 Agent

requirement_analyst
  • 理解中文业务请求
  • 提取主题、时间、指标、维度
  • 输出结构化需求
02

报表 SQL 开发 Agent

report_sql_developer
  • 基于需求和数据库上下文生成 SQL
  • 设计报表数据块
  • 按审核建议修正 SQL
03

数据平台 Agent

data_platform_operator
  • 读取 schema、指标口径和报表目录
  • 执行只读 SELECT 查询
  • 返回结果或数据库错误
04

报表审核 Agent

report_reviewer
  • 检查 SQL 是否成功执行
  • 检查结果是否为空或字段缺失
  • 不通过时给出修正建议
05

HTML 报告生成 Agent

report_html_generator
  • 接收通过审核的数据
  • 生成完整 HTML 报告
  • 不编造查询结果中不存在的数据

2. 运行流程

主对话把产物逐步转交给下一个角色，审核失败时回到 SQL 开发 Agent 修正。

1

用户提出需求

自然语言描述要看的报表主题和指标。

2

需求结构化

得到主题、时间范围、指标、维度、输出格式。

3

读取数据库上下文

获取 mart schema、指标口径、语义视图和报表目录。

4

生成 SQL 与报表方案

产出只读 SQL 和要展示的数据块。

5

执行只读查询

通过受控访问入口执行 SELECT / WITH 查询。

6

结果审核

检查执行状态、空结果、字段完整性和口径匹配。

7

失败则反馈

返回 SQL 开发 Agent 修正，建议最多重试 2 次。

8

通过则生成报告

报告生成 Agent 产出完整 HTML。

9

保存并返回路径

Main Agent 保存到 reports/ 并返回文件路径。

反馈闭环

  • SQL 错误、空结果、字段缺失或口径不匹配时，审核 Agent 不让流程进入报告生成。
  • Main Agent 把审核意见交回 SQL 开发 Agent，修正后再交给数据平台 Agent 查询。

数据平台边界

Agent 不直接无限制访问数据库，而是复用项目已有访问层。
  • python -m app.demo_run inspect 读取数据库上下文
  • python -m app.demo_run query --sql "..." 执行只读查询
  • app/db/executor.py 拦截写操作并限制单条 SQL

3. 核心产物

每一步只传递下一步需要的内容，避免 Main Agent 上下文膨胀。

结构化需求 标题、时间范围、指标、维度、筛选条件、输出格式。
数据库上下文 schema、字段、指标定义、语义视图、健康检查。
SQL 与查询结果 只读 SELECT / WITH SQL，返回 JSON 行数据和执行状态。
HTML 报告 保存在 reports/，供业务用户直接打开查看。
